3.@RequestMapping注解

1.标记地点

SpringMVC使用@RequestMapping注解为控制器指定可以处理哪些 URL 请求

  1. 在控制器的类定义及方法定义处都可标注 @RequestMapping

（1）标记在类上：提供初步的请求映射信息。相对于 WEB 应用的根目录

（2）标记在方法上：提供进一步的细分映射信息。相对于标记在类上的 URL。标记在方法上的此注解，不可重复，但是如果其中一个方法，对应的类有一层mapping了，再重复写就也是可以的，因为全路径就不重复了

若类上未标注 @RequestMapping，则方法处标记的 URL 相对于 WEB 应用的根目录

作用：DispatcherServlet 截获请求后，就通过控制器上 @RequestMapping 提供的映射信息确定请求所对应的处理方法。

2.其他属性详解

@RequestMapping：

​ method:可以来限定请求方式，不是规定的请求方式就报错

​ HTTP协议中规定的请求方式

​ 比如method=RequestMethod.POST ;就是只接收这种类型的请求

​ 默认是全接收

​ params:规定请求参数

​ params 和 headers支持简单的表达式：

​ param1: 表示请求必须包含名为 param1 的请求参数

​ eg : params = {“username1”} 请求参数中必须包含username1这个参数

​ !param1: 表示请求不能包含名为 param1 的请求参数

​ 带了就会报错

​ param1 != value1: 表示请求包含名为 param1 的请求参数，但其值不能为 value1

​ {“param1=value1”, “param2”}: 请求必须包含名为 param1 和param2 的两个请求参数，且 param1 参数的值必须为 value1—多规则

​ headers: 规定请求头，表达式也和params一样

​ 需要是请求头里的任意字段

​ consumes: 只接收内容类型是哪种的请求，规定请求头中的Content-Type

​ produces: 告诉浏览器返回的内容类型是什么，给响应头中加上Content-Type:text/html

5.Rest 风格URL路径

核心思想：URL路径高度统一，按照提交请求的方式不同划分不同的操作内容

就是 HTTP 协议里面，四个表示操作方式的动词：GET、POST、PUT、DELETE。它们分别对应四种基本操作：GET 用来获取资源，POST 用来新建资源，PUT 用来更新资源，DELETE 用来删除资源。

问题在于，页面上只能发起GET POST请求，其他的请求无法执行

•示例：

–/order/1 HTTP GET ：得到 id = 1 的 order

–/order/1 HTTP DELETE：**删除 id = 1的 order

–/order/1 HTTP PUT：更新id = 1的 order

–/order HTTP POST：新增 order

如何从页面发起PUT DELETE 形式的请求呢？

  1. HiddenHttpMethodFilter：浏览器 form 表单只支持 GET 与 POST 请求，而DELETE、PUT 等 method 并不支持，Spring3.0 添加了一个过滤器，可以将这些请求转换为标准的 http 方法，使得支持 GET、POST、PUT 与 DELETE 请求。

  2. 按照以下要求：

    1. 创建一个post类型的表单

    2. 表单项中携带一个_method的参数

    3. 这个_method的值就是DELETE、PUT

高版本tomcat只支持post，get请求，处理方式是在jsp头添加 isErrorPage=”true” 即可

21.拦截器

SpringMVC提供了拦截器机制；允许运行目标方法之前进行一些拦截器操作

和Filter相似，但更加强大

HandlerInterceptor : SpringMVC

因为它有三个方法

preHandle():在目标方法运行之前调用，返回boolean；返回true 放行，返回false，不放行

postHandle() 在目标方法运行之后调用。紧接着

afterCompletion ()：在整个请求完成之后，来到目标页面，资源响应之后

拦截器是一个接口

1.实现拦截器需要实现HandlerInterceptor接口

2.在SpringMVC配置文件中注册这个拦截器的工作

​ 配置这个拦截器拦截哪些请求的目标方法

配置多个拦截器就正常在<mvc: interceptors>里面写就可以了

其他流程

  1. 只要preHandle不放行，就没有以后的流程

  2. 只要放行了，就算会报错，afterCompletion() 内的一定会执行

  3. 多个拦截器并存：

    1. 正常流程：先配置的先执行，同时preHandle()顺序执行，postHandle() 和afterCompletion ()逆序执行

    2. 异常流程：

      1.不放行：哪一块不放行，从此以后都没有，但是已经放行的拦截器还会执行afterCompletion()里的方法

基本流程分析：

preHandle执行，里面是一个遍历的方法，拿到每一个拦截器，都执行pre方法，一旦有一个执行为false，直接返回false，并记录一下索引，用于判断后面的哪个拦截器的afterCom…会执行

如果拦截器A 已经被放行，那么之后不管是执行错误，还是被第二个拦截器拦下，都会执行它，因为哪里都有它

什么时候用拦截器，什么时候用Filter

如果写某些功能需要其他组件配合使用，我们就写拦截器，其他情况可以写Filter

23.SpringMVC详细运行流程

  1. 所有请求，前端控制器(DispatcherServlet)收到请求，调用doDispatcher进行处理
  2. 根据HandlerMapping中保存的请求映射信息找到，处理当前请求的处理器执行链（包含拦截器）
  3. 根据当前处理器找到它的HandlerAdapter（适配器）
  4. 拦截器的preHandle先执行
  5. 适配器执行目标方法
    1. ModelAttribute注解标注的方法提前运行
    2. 执行目标方法的时候（确定目标方法用的参数）
      1. 有注解就按照注解的参数注入
      2. 没注解
        1. 看看是否是Model、Map类型以及其他的
        2. 如果是自定义类型
          1. 从隐含模型中看看有没有，有就从隐含模型中拿
          2. 如果没有，再看SessionAttributes标注的属性，如果是的话就拿，拿不到就抛异常
          3. 都不是，利用反射创建新的对象
  6. 拦截器的postHandle执行
  7. 处理视图处理结果，页面渲染流程
    1. 如果有异常，使用异常解析器处理异常，处理完后还会返回ModelAndView
    2. 调用render进行渲染
      1. 视图解析器根据视图名得到视图对象
      2. 视图对象调用render方法
    3. 视图拦截器的afterCompletion

控制反转（Inversion of Control，缩写为IoC），是面向对象编程中的一种设计原则，可以用来减低计算机代码之间的耦合度。其中最常见的方式叫做依赖注入（Dependency Injection，简称DI），还有一种方式叫“依赖查找”（Dependency Lookup）。通过控制反转，对象在被创建的时候，由一个调控系统内所有对象的外界实体将其所依赖的对象的引用传递给它。也可以说，依赖被注入到对象中。

依赖注入一共有三种方式：

​ 1.构造函数注入

​ 2.set方法注入

​ 3.使用注解

注解注入

注解分为四类：

1.用于创建对象的:他们的作用和在xml编写一个bean标签是一样的，出现位置在类上

  • @Component:

    作用：用于把当前类对象存入spring容器中

    属性：

    value：用于指定bean的id。当我们不写时，它的默认值是当前类名，且首字母是小写

  • @Controller：表现层

  • @Service：业务层

  • @Repository：持久层

    这三个注解的作用和属性与Component一模一样，他们三个是spring框架为我们提供明确的三层使用的注解，使我们的三层对象更加清晰

2.用于注入数据的：他们的作用就和 property 标签类似

  • @Autowired

    作用：自动按照类型注入。只要容器中有唯一的一个bean对象类型和要注入的变量类型匹配，就可以注入

    如果IOC容器中没有任何的bean类型，就会报错

    如果IOC容器中有多个匹配值，先按照类型圈定出匹配的对象

    使用变量名称来作为第二重筛选条件，继续查找，如果变量名和map中的名称也不相同，就报错

    出现位置：可以是变量上，也可以是方法上（最常用的就是这两种）

    由于这个注入方式是根据bean对象类型的，在spring的IOC MAP中，只要存在着对应的接口或者继承了接口的实现类即可

    把实现类的注解去掉，或者去掉implements 都不能成功注入

  • @Qualifier:

    作用：在按照类注入的基础之上，再按照名称注入。它在给类成员变量注入时不可以单独使用。但给方法参数注入时可以

    属性:value:用于指定被注入类的id

  • @Resource 为什么这个注解不在spring包里，它在javax.annotation里

    作用：直接按照bean的id注入，它可以独立使用

    以上三种注解都只能注入其他bean类型的数据，而基本类型和String类型无法使用上述注解

    另外，集合类型的注入只能通过xml

  • @Value

    作用：用于注入基本类型的String类型的数据

    属性：value:用于指定数据的值，它可以使用spring中SpEL（也就是spring的EL表达式）

    SpEL写法:#{表达式}

    下面这个只是恰好长得像el表达式，但它不是el表达式

3.用于改变作用范围的：他们的作用就和bean标签的scope属性一样

  • @Scope

    作用：用于指定bean的作用范围,写在类上

    属性:singleton prototype 默认是singleton

4.生命周期相关的：他们的作用就和bean标签的init-method destory-method一样的
  • @PreDestory : 用于指定销毁方法 写在方法上，一销毁，这个方法就执行—-多例对象spring不负责销毁
  • @PostConstruct： 用于指定初始化方法 同上
